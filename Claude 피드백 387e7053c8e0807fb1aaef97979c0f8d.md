# Claude 피드백

## Manyfast AI Coach 프로토타입 감사 리포트

제품 구조 파악 요약: **PRD → 기능명세서 → 유저플로우** 3탭 기획 도구에, 왼쪽 Manny AI Assistant 사이드바 + 오른쪽 AI Decision Coach 패널이 붙은 구조. 코드 2293줄 전부 단일 App.tsx에 집중.

---

### Top 10 Critical Issues

**1. AI Coach가 목업 데이터만 반환 (가장 심각)**

Problem: `/api/coach`와 `/api/chat` 엔드포인트가 실제로 없음. 에러 시 `getDynamicAiResponse()`라는 하드코딩된 fallback이 동작하는데, 이건 AI가 아니라 `includes()` 키워드 매칭.

Impact: 면접에서 "AI가 실제로 분석한 건가요?"라고 물으면 대답이 곤란해짐. 코치의 핵심 가치인 신뢰성이 무너짐.

Evidence: `getDynamicAiResponse()` 함수가 input.toLowerCase()로 키워드 매칭. `coachAnalysis`는 하드코딩된 초기값 그대로 렌더링됨.

Recommendation: Anthropic API를 직접 호출하는 `/api/coach` 엔드포인트 연결. 이미 프로젝트가 AI Studio 기반이라 Gemini API 호출 코드 붙이기 어렵지 않음.

Priority: P0 / Severity: **Critical**

---

**2. 추천 근거의 신뢰성 없음 (Layer 3)**

Problem: 벤치마킹 탭의 팩트 데이터 출처가 "Industry heuristic (unverified)", "User-provided"로 표기됨. 클릭 가능한 레퍼런스 없음.

Impact: "출처검증됨(Verified)" 뱃지가 붙은 추천이 실제로는 검증 안 된 내용임. 신뢰도를 역으로 깎음.

Evidence: `source: "Industry heuristic (unverified)"` 인데 UI에선 `출처검증됨` 뱃지가 표시되는 로직 있음.

Recommendation: 신뢰도 레이블을 실제 데이터 유형에 맞게 정직하게 표시. Lean Startup, The Mom Test 같은 프레임워크 이름 + 인용 페이지 직접 명기.

Priority: P0 / Severity: **Critical**

---

**3. 왼쪽 Manny AI가 GPT-4o라고 표기하나 실제론 아님**

Problem: 채팅 입력창에 "gpt-4o" 라벨이 하드코딩되어 있고, 왼쪽 사이드바 배지도 "gpt-4o". 실제 호출은 없거나 fallback.

Impact: 데모 시 신뢰도 즉시 붕괴. 포트폴리오에서 명백한 허위 표시.

Recommendation: 실제 사용 모델명으로 교체. Gemini Flash든 Claude Sonnet이든 실제로 연결된 걸 표기.

Priority: P0 / Severity: **Critical**

---

**4. "즉시 기획 문서 반영하기" 버튼이 alert()로 동작**

Problem: `handleApplyRecommendation()`이 `alert()`를 씀. 브라우저 기본 알림창이 뜸.

Impact: 프로토타입의 완성도 인상을 한 번에 깎는 요소. PM 포트폴리오에서 치명적인 UX 결함으로 읽힘.

Recommendation: toast 알림 또는 인라인 확인 UI로 교체. shadcn/ui의 Toast 컴포넌트 쓰면 10분 작업.

Priority: P0 / Severity: **Critical**

---

**5. "새 기능/요구사항 추가"가 prompt()로 동작**

Problem: `handleAddNewAc()`, `handleAddNewFeature()`, `handleAddNewRequirement()` 모두 `prompt()` 사용.

Impact: 브라우저 기본 입력창은 커스텀 스타일링 불가. 전체 디자인 시스템과 완전히 동떨어짐. 면접관 데모 시 인상 나쁨.

Recommendation: 인라인 입력 필드 또는 모달로 교체.

Priority: P1 / Severity: **Major**

---

**6. AI Coach가 컨텍스트를 진짜로 읽지 않음 (Layer 2)**

Problem: 탭을 바꾸거나 기능을 선택해도 `triggerCoachAnalysis()`가 fallback mock data만 반환. API 실패 시 coachAnalysis 상태가 그대로 유지되어 마치 실시간 분석처럼 보이지만 실제론 초기값 고정.

Impact: "Context Awareness"가 0. PRD탭에서도, 기능명세서탭에서도 같은 코멘트가 나옴.

Recommendation: 최소한 mock이더라도 탭별로 다른 분석 결과를 보여주는 컨텍스트 스위칭 구현.

Priority: P1 / Severity: **Major**

---

**7. 마인드맵 뷰의 설명이 하드코딩 (Layer 2)**

Problem: 마인드맵에서 노드를 클릭해도 오른쪽 패널 설명이 "매칭 알고리즘과 수용 가치 매니패스트는..."으로 고정. 어떤 노드를 눌러도 동일한 텍스트.

Recommendation: 각 Requirement의 features 목록에서 동적으로 설명 렌더링.

Priority: P1 / Severity: **Major**

---

**8. 스타트업 검증 챌린지가 약함 (Layer 8)**

Problem: Coach 리스크 탭의 챌린지 질문이 WTP 하나뿐. 타겟 사용자가 "모든 직장인"이라는 과도하게 넓은 가정에 대한 챌린징 없음. Mom Test 원칙에 따른 "왜 지금 이 문제를 해결하려 하는가?" 수준의 질문 부재.

Recommendation: 챌린지 항목을 최소 5개로 확장. 타겟 과도 확대, WTP, 경쟁사 차별화, 핵심 지표 검증 가능성 등 포함.

Priority: P2 / Severity: **Major**

---

**9. 코드 구조 — 2293줄 단일 파일 (유지보수성)**

Problem: 모든 로직이 App.tsx 하나에 있음. 타입, 상수, 컴포넌트, 핸들러, API 호출이 뒤섞임.

Impact: 포트폴리오로는 괜찮지만, "PM이 직접 코드를 짰다"는 인상을 주려면 구조에 대한 설명 준비가 필요. 면접에서 "왜 이렇게 구성했나요?"가 나올 수 있음.

Recommendation: 당장 리팩토링보다는 README에 "AI Studio 프로토타이핑 목적으로 단일 파일 구조를 선택했다"고 명시.

Priority: P2 / Severity: **Minor**

---

**10. 유저플로우 레인 C가 빈 화면**

Problem: "커피챗 일정 약속 조율 캘린더 구간" 레인이 "추후 추가 예정" 텍스트만 있음.

Impact: 프로토타입 완성도 인상 저하. 핵심 기능인 커피챗 스케줄링 플로우가 없음.

Recommendation: 목업 수준이라도 캘린더 선택 → 상대방 슬롯 제안 → 확정 3단계 노드 추가.

Priority: P2 / Severity: **Major**

### Top 10 High-Impact Improvements

1. **실제 AI API 연결** — Gemini Flash나 Claude Sonnet을 `/api/coach`에 연결. 이게 되면 나머지 절반의 문제가 자동 해결됨.
2. **alert/prompt → 인라인 UI** — toast + 인라인 입력 필드로 교체. 반나절 작업.
3. **신뢰도 레이블 정직하게** — "출처검증됨" 뱃지 기준을 실제 유형에 맞게 재정의.
4. **컨텍스트별 코치 응답 분리** — 탭마다 다른 mock 데이터라도 준비.
5. **마인드맵 노드별 동적 설명** — 하드코딩 제거, 데이터 바인딩.
6. **유저플로우 레인 C 채우기** — 커피챗 예약 플로우 최소 3노드.
7. **챌린지 질문 5개 이상** — Mom Test 기반으로 확장.
8. **모델명 정직하게 표기** — gpt-4o 레이블 제거, 실제 모델 표기.
9. **PRD 섹션 편집 기능** — 현재 PRD는 읽기 전용. 직접 수정 가능하게.
10. **README 업데이트** — 프로젝트 목적, 구조 선택 이유, 사용 기술 스택 명시.

### 최종 판단

**"이 AI Coach가 원래 제품만 사용하는 것보다 PM의 의사결정을 실질적으로 개선하는가?"**

**현재 상태에서는 No.** 이유는 하나: AI가 실제로 분석하지 않기 때문. 코치 패널에 표시되는 내용은 초기 하드코딩값이고, 채팅 응답은 키워드 매칭 fallback.

단 **구조와 UX 프레임은 Yes를 향하고 있음.** 3탭 구조, 컨텍스트 패싱 로직(`payload` 객체), AC 자동 반영 기능, 리스크/벤치마킹 분리 탭 — 이 뼈대는 잘 설계됐어. API 연결만 되면 즉시 가치가 올라감.
